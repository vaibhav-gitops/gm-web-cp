    Introducing Gitmoxi

    Gitmoxi is a continuous deployment (CD) product that uses GitOps paradigm to deploy services to Amazon Elastic Container Service (ECS) with AWS Fargate. ECS service deployment related files such as service and task definition are maintained in Git. When these files are changed and committed, Gitmoxi triggers the creation or update of ECS services based on the changes. Several services across multiple clusters and regions can be deployed simultaneously by changing the associated Git files and committing them to configured repositories and branches.

    Benefits

    There are several benefits of Gitmoxi continuous deployment approach for ECS:

    • Git collaboration and version control - Service deployment related assets such as service and task definitions are version controlled and tracked in Git. This enables platform, devops, cloud infrastructure, and application development teams to collaborate seamlessly using the beloved Git features such as branching, reviews, approvals. Storing these assets in Git also helps with root-cause analysis, forensics, and diagnostics to address the quintessential question of "what changed" in case of failures.

    • Decouple service and task from infra-as-code - Gitmoxi approach decouples the service and task definition from infra-as-code. Service and task definitions change quite frequently. If they are baked into Terraform, CDK, or CloudFormation, then changes to service and task attributes ends up causing entire stacks to go through unnecessary update evaluation, which often causes unexpected errors. With Gitmoxi you can simply provide the relevant output from infra-as-code such as subnets, IAM roles, security groups, load balancers as a reference input file, and Gitmoxi will add those to service and task definition before deployment. With this decoupling, your service can change freely, frequently, and worry-free without impacting infra-as-code.

    • Support for multiple deployment paradigms - Gitmoxi has its own deployment controller which supports both blue-green deployments and rolling update. With blue-green three types of traffic shifting patterns are supported - all-at-once, linear, and canary. For rolling update, Gitmoxi supports using ECS built-in deployment along with all of its features.

    • Flexible deployment circuit breaking and rollback - We understand that every service is different. Some may have a lot of replica tasks, take long to start, or require rollback on very low failure counts. Gitmoxi provides customizable deployment circuit breaking based on a combination of timeouts and failure thresholds that can be tailored to meet your application needs.

    • Comprehensive error reporting - Deployments can fail for multiple reasons across the multiple steps and multiple resources involved. Service or task definition could have incorrect/missing parameters, containers could be failing health checks, or CloudWatch alarms could be indicating errors in new version. Gitmoxi diligently gathers the errors throughout the deployment process and reports them in user-friendly UI.

    • Push or pull or both - With Gitmoxi you can push the commit changes and invoke the deployment, for example, using Github Webhook posts or explicitly call the deploy API in a pipeline. Or, you can configure the Gitmoxi poller to periodically check for latest commit and apply the changes. Or, you can use both! Gitmoxi poller can serve as a safety net in case the push approach misses any commits.

    How does it work?

    Assume that you have created an ECS service and task definition JSON file. You put these files in a Git repository. In Gitmoxi, you configure the repository, branch, and Git access token. Now, whenever the ECS service and task definition files are changed and committed to the configured branch, Gitmoxi can either receive a GitHub webhook and/or poll for the commit changes. Gitmoxi deployment controllers will then appropriately create or update the ECS service. Few additional files can be provided such as a -/ deployment configuration: to setup aspects such as traffic shifting and circuit breaking, and -/ input file: to provide input such as subnets, load balancers from other sources such as output of an infra-as-code tool. Your continuous integration (CI) process can update the container image or other attributes in the service and task definition files, commit them to Git repository, and trigger the Gitmoxi deployments. See Getting Started Guide for more details.
